# Todolist App Wireframe

이 문서는 `docs/1-prd.md`, `docs/2-erd.md`, `docs/3-api-spec.yaml`을 기준으로 도출한 텍스트 와이어프레임이다.

## 화면 도출 기준

- PRD의 핵심 사용자 스토리에서 직접 필요한 화면만 우선 반영한다.
- 초기 PRD는 비로그인 + 로컬 저장이므로, 로그인 화면은 필수 화면이 아니라 서버 모드 선택 화면으로 본다.
- ERD의 `todos`, `tags`, `todo_tags`를 중심으로 목록, 생성/수정, 삭제, 태그 관리 화면을 구성한다.
- API 명세의 `GET /todos`, `POST /todos`, `PATCH /todos/{todoId}`, `DELETE /todos/{todoId}`, `PATCH /todos/{todoId}/completion`, `GET /tags`, `POST /tags`, `PATCH /tags/{tagId}`, `DELETE /tags/{tagId}`를 화면 설계의 기준으로 사용한다.

## 공통 레이아웃 규칙

- 상단 고정 영역: 앱 제목, 현재 모드 표시, 주요 액션 버튼
- 본문 영역: 목록 또는 폼 콘텐츠
- 보조 영역: 필터, 태그 선택, 상세 정보
- 모바일: 단일 컬럼, 하단 시트 또는 전면 모달 우선
- 태블릿: 단일 컬럼을 유지하되 폼과 필터를 2열로 재배치 가능
- 데스크톱: 좌측 목록, 우측 상세/편집 패널의 분할 구조 가능

### 사용자 인터랙션 흐름

1. 사용자가 앱을 열면 기본적으로 오늘/전체 목록이 보인다.
2. 검색어나 필터를 변경하면 목록이 즉시 갱신된다.
3. 제목을 빠르게 입력해 `빠른 추가`를 저장하면 새 할 일이 목록에 추가된다.
4. 체크박스를 누르면 `PATCH /todos/{todoId}/completion`으로 완료/미완료가 전환된다.
5. 수정 버튼을 누르면 2번 화면의 편집 패널이 열린다.
6. 삭제 버튼을 누르면 3번 화면의 확인 다이얼로그가 뜬다.

### 반응형 레이아웃 변화

- 모바일 `360-767px`
  - 상단 제목과 `+추가` 버튼만 1차 노출한다.
  - 필터는 접이식 섹션 또는 하단 시트로 전환한다.
  - 빠른 추가는 상단 고정 또는 하단 전용 입력 영역으로 배치한다.
  - 목록 항목은 한 줄 요약 + 두 줄 보조 정보 구조로 축소한다.
- 태블릿 `768-1023px`
  - 검색과 필터를 2열 또는 칩형 레이아웃으로 배치한다.
  - 빠른 추가 폼을 목록 상단에 펼친 상태로 유지할 수 있다.
  - 목록은 넓게 보여주되 액션 버튼은 우측 정렬한다.
- 데스크톱 `1024px+`
  - 좌측 목록, 우측 상세/편집 패널 분할이 가능하다.
  - 필터 바를 상단 고정형 툴바로 유지한다.
  - 페이지네이션과 빈 상태를 목록 하단에 안정적으로 배치한다.

## 2. 할 일 추가/수정 화면

### 근거가 되는 사용자 스토리

- 3.1 할 일 추가
- 3.2 할 일 수정

### 화면 목적

- 사용자가 제목, 마감일, 우선순위, 태그를 입력하거나 수정한다.
- 완료 상태는 이 화면이 아니라 목록 화면의 체크박스 액션으로 관리한다.

### ASCII 레이아웃 스케치

```text
┌────────────────────────────────────────────┐
│ [새 할 일 / 할 일 수정]                   │
├────────────────────────────────────────────┤
│ 제목*                                      │
│ [........................................] │
│                                            │
│ 마감일                                     │
│ [날짜 선택▼]                               │
│                                            │
│ 우선순위                                   │
│ ( ) 없음  ( ) 낮음  ( ) 보통  ( ) 높음     │
│                                            │
│ 태그                                       │
│ [업무 x] [개인 x] [+ 태그 추가]            │
│                                            │
│ [취소]                           [저장]    │
└────────────────────────────────────────────┘
```

### 호출하는 API 엔드포인트

- `GET /todos/{todoId}`: 수정 진입 시 기존 값 로딩
- `POST /todos`: 새 할 일 생성
- `PATCH /todos/{todoId}`: 기존 할 일 수정
- `GET /tags`: 태그 선택 목록 조회
- `POST /tags`: 태그가 없을 때 즉시 생성

### 컴포넌트 트리

- `TodoEditorPanel`
  - `EditorHeader`
  - `TodoForm`
    - `TitleField`
    - `DueDateField`
    - `PriorityField`
    - `TagSelector`
      - `SelectedTagChips`
      - `TagDropdown`
      - `CreateTagAction`
    - `FormActions`
      - `CancelButton`
      - `SaveButton`
  - `FieldErrorList`

### 사용자 인터랙션 흐름

1. 사용자가 `+추가` 또는 목록의 `수정` 버튼을 누른다.
2. 추가 모드면 빈 폼, 수정 모드면 `GET /todos/{todoId}` 결과를 채운 폼이 열린다.
3. 제목은 필수 입력으로 검증한다.
4. 태그는 기존 태그 선택 또는 새 태그 생성으로 처리한다.
5. 저장 성공 시 화면을 닫고 목록을 갱신한다.
6. 취소 시 변경 내용을 저장하지 않고 닫는다.

### 반응형 레이아웃 변화

- 모바일 `360-767px`
  - 전면 화면 또는 하단 시트로 열린다.
  - 저장/취소 버튼은 하단 고정으로 둔다.
  - 태그 선택은 바텀시트 또는 풀스크린 선택기로 확장한다.
- 태블릿 `768-1023px`
  - 중앙 모달 또는 우측 패널로 표시한다.
  - 입력 간격을 넓히고 태그 선택을 펼침형으로 둔다.
- 데스크톱 `1024px+`
  - 목록 오른쪽 인라인 패널로 열 수 있다.
  - 목록과 폼을 동시에 보며 수정할 수 있다.
  - 긴 설명 대신 즉시 입력 가능한 레이아웃을 유지한다.

### 화면 목적


### 호출하는 API 엔드포인트

- `DELETE /todos/{todoId}`

### 컴포넌트 트리

- `DeleteConfirmDialog`
  - `DialogTitle`
  - `TodoSummary`
  - `WarningText`
  - `DialogActions`
    - `CancelButton`
    - `DeleteButton`

### 사용자 인터랙션 흐름

1. 사용자가 목록에서 `삭제`를 클릭한다.
2. 대상 할 일의 제목과 태그 요약을 보여준다.
3. 사용자가 `취소`를 누르면 아무 변화 없이 닫힌다.
4. 사용자가 `삭제`를 누르면 `DELETE /todos/{todoId}`를 호출한다.
5. 성공 시 다이얼로그를 닫고 목록에서 항목을 제거한다.

### 반응형 레이아웃 변화

- 모바일 `360-767px`
  - 하단 시트 형태가 적합하다.
  - `삭제` 버튼은 위험색으로 크게 배치한다.
- 태블릿 `768-1023px`
  - 중앙 모달로 표시한다.
  - 대상 요약을 한 번에 읽을 수 있도록 간격을 넓힌다.
- 데스크톱 `1024px+`
  - 짧은 중앙 모달이 적합하다.
  - 키보드 포커스 트랩과 ESC 취소를 적용한다.

```text
┌──────────────────────────────────────────────┐
```

### 호출하는 API 엔드포인트

- `GET /tags`
- `POST /tags`
- `PATCH /tags/{tagId}`
- `DELETE /tags/{tagId}`

### 컴포넌트 트리

- `TagManagementPage`
  - `TagSearchField`
  - `TagCreateForm`
    - `TagNameInput`
    - `AddTagButton`
  - `TagList`

      - `TagName`

### 사용자 인터랙션 흐름

1. 사용자는 태그 목록을 확인하거나 검색한다.
2. 새 태그를 입력해 `POST /tags`로 생성한다.

4. 삭제 시 `DELETE /tags/{tagId}`를 호출하고 연결된 할 일의 태그 관계는 ERD의 `ON DELETE CASCADE`로 정리된다.
5. 태그 관리 결과는 할 일 편집 폼의 태그 선택 목록에 즉시 반영된다.

### 반응형 레이아웃 변화

- 모바일 `360-767px`
  - 별도 전체 화면 또는 설정 하위 화면으로 연다.
  - 생성/수정 입력은 인라인보다 하단 시트를 선호한다.
- 태블릿 `768-1023px`
  - 목록과 생성 폼을 세로로 배치한다.
  - 편집은 인라인으로 간단히 처리한다.
- 데스크톱 `1024px+`
  - 좌측 태그 목록, 우측 태그 생성/수정 패널이 가능하다.
  - 태그가 많을 때 검색과 정렬을 함께 제공할 수 있다.

## 5. 로그인 화면

### 근거가 되는 사용자 스토리

- PRD 기준 필수 화면은 아님
- API 명세의 JWT Bearer 인증을 사용하는 서버 모드에서만 필요

### 화면 목적

- 서버 모드를 사용할 때만 사용자 인증을 수행한다.
- PRD의 로컬 저장 모드에서는 이 화면이 필요하지 않다.

### ASCII 레이아웃 스케치

```text
┌─────────────────────────────┐
│ 로그인                      │
├─────────────────────────────┤
│ 이메일                      │
│ [........................]  │
│ 비밀번호                    │
│ [........................]  │
│ [로그인]                    │
│ [오류 메시지]               │
└─────────────────────────────┘
```

### 호출하는 API 엔드포인트

- `POST /auth/login`
- `POST /auth/refresh`
- `POST /auth/logout`

### 컴포넌트 트리

- `LoginPage`
  - `LoginForm`
    - `EmailField`
    - `PasswordField`
    - `LoginButton`
    - `AuthErrorMessage`

### 사용자 인터랙션 흐름

1. 사용자가 이메일과 비밀번호를 입력한다.
2. 로그인 성공 시 토큰을 저장하고 목록 화면으로 이동한다.
3. 만료 시 `POST /auth/refresh`로 재발급한다.
4. 로그아웃 시 `POST /auth/logout`을 호출한다.

### 반응형 레이아웃 변화

- 모바일 `360-767px`
  - 단일 폼 화면을 유지한다.
  - 입력과 버튼을 세로로 배치한다.
- 태블릿 `768-1023px`
  - 중앙 카드 형태를 사용한다.
- 데스크톱 `1024px+`
  - 중앙에 고정 폭 카드, 배경은 앱 목록과 분리된 단순 레이아웃이 적합하다.

### 사용자 인터랙션 흐름

1. 데이터가 없으면 첫 할 일 추가 CTA를 보여준다.
2. 검색 또는 필터 결과가 없으면 현재 조건을 초기화하도록 유도한다.
3. 에러가 나면 재시도 버튼으로 목록을 다시 조회한다.

### 반응형 레이아웃 변화

- 모바일 `360-767px`: 중앙 정렬 단일 블록
- 태블릿 `768-1023px`: 카드 폭을 넓히고 버튼 간격을 늘림
- 데스크톱 `1024px+`: 목록 영역 안의 보조 상태로 작게 표시

## 7. 화면 간 전환 요약

- `할 일 목록 화면`에서 `+추가`를 누르면 `할 일 추가/수정 화면`이 열린다.
- `할 일 목록 화면`에서 `수정`을 누르면 해당 항목을 채운 편집 화면이 열린다.
- `할 일 목록 화면`에서 `삭제`를 누르면 `삭제 확인 다이얼로그`가 열린다.
- `할 일 추가/수정 화면`에서 저장하면 목록 화면으로 돌아간다.
- `태그 관리 화면`에서 생성/수정/삭제한 결과는 목록 화면과 편집 화면의 태그 선택기에 즉시 반영된다.
- `로그인 화면`은 서버 모드에서만 진입하고, 로컬 모드에서는 건너뛴다.

## 8. 구현 메모

- 목록 화면이 가장 중요한 허브 화면이다.
- 모바일에서는 전면 패널, 데스크톱에서는 분할 패널로 같은 기능을 제공하는 것이 가장 효율적이다.
- PRD 초기 범위만 구현한다면 로그인 화면은 제외해도 된다.
- API 명세를 그대로 사용한다면 서버 모드와 로컬 모드를 분리하는 진입 전략이 필요하다.
