# Todolist App Execution Plan

이 실행계획은 `docs/1-prd.md`, `docs/2-erd.md`, `docs/3-api-spec.yaml`, `docs/5-wireframe.md`, `docs/5-1-style-guide.md`를 기준으로 작성했다.  
PRD는 초기 범위를 로컬 저장 중심으로 정의하지만, 본 계획은 요청된 구현 경로에 맞춰 `Supabase + Express + React + TailwindCSS + Vercel` 조합으로 진행한다.

---

## Task 0: 환경 세팅

### 목표

프로젝트가 프론트엔드, 백엔드, DB 작업을 병렬로 진행할 수 있는 상태가 되도록 기본 구조와 개발 환경을 정리한다.

### 세부 작업 순서

1. 저장소 루트에 `apps/frontend`, `apps/backend`, `packages/shared` 구조를 만든다.
2. 프론트엔드용 React 프로젝트를 생성한다.
3. 백엔드용 Express 프로젝트를 생성한다.
4. 공통 타입과 유틸을 둘 `packages/shared`를 준비한다.
5. TailwindCSS, PostCSS, Autoprefixer를 프론트엔드에 설치한다.
6. Express 서버에 필요한 기본 미들웨어를 정리한다.
7. Supabase 프로젝트와 연결할 `.env` 템플릿을 만든다.
8. `.env.example`에 필수 환경변수를 정의한다.
9. 로컬 개발 포트와 실행 스크립트를 표준화한다.
10. ESLint, Prettier, TypeScript 설정을 통일한다.
11. 기본 폴더별 README 또는 실행 가이드를 추가한다.

### 예상 소요 시간

- 4 ~ 6시간

### 완료 기준 (Definition of Done)

- 프론트엔드와 백엔드가 각자 독립적으로 실행된다.
- `.env.example`만 보고 필요한 환경변수를 알 수 있다.
- 공통 타입과 디렉터리 구조가 이후 작업에 방해되지 않는다.
- 로컬 개발 실행 명령이 문서화되어 있다.

### 다음 Task로 넘어가기 전 확인사항

- 프론트엔드와 백엔드가 모두 `npm run dev` 또는 동등한 명령으로 실행되는가
- Supabase 연결 정보가 `.env.example`에 반영되었는가
- 코드 스타일과 린트 규칙이 양쪽 프로젝트에서 일치하는가

## Task 1: DB & 백엔드

### 목표

Supabase에 DB 테이블을 만들고, Express 서버에서 PRD의 할 일/태그 기능을 제공하는 API를 구현한다.

### 세부 작업 순서

1. Supabase에 프로젝트를 생성하고 로컬/스테이징/프로덕션 분리 전략을 정한다.
2. `todos`, `tags`, `todo_tags` 테이블과 제약조건을 생성한다.
3. 인덱스와 `ON DELETE CASCADE`를 적용한다.
4. 필요한 시드 데이터 또는 초기 마이그레이션을 작성한다.
5. Express 서버의 기본 라우팅 구조를 만든다.
6. API 스펙의 request/response shape와 동일한 DTO를 정의한다.
7. validation, error handler, not-found handler를 구현한다.
8. `GET /todos`, `POST /todos`, `GET /todos/{todoId}`를 구현한다.
9. `PATCH /todos/{todoId}`, `PATCH /todos/{todoId}/completion`, `DELETE /todos/{todoId}`를 구현한다.
10. `GET /tags`, `POST /tags`, `PATCH /tags/{tagId}`, `DELETE /tags/{tagId}`를 구현한다.
11. JWT Bearer 인증 미들웨어를 연결한다.
12. 인증이 필요한 라우트에 `security` 요구사항을 반영한다.
13. DB 접근 레이어를 분리해 라우트와 비즈니스 로직을 분리한다.
14. Swagger/OpenAPI 문서와 실제 응답 구조를 맞춘다.

### 예상 소요 시간

- 10 ~ 16시간

### 완료 기준 (Definition of Done)

- Supabase 테이블이 문서의 ERD와 일치한다.
- 모든 API 엔드포인트가 문서화된 스펙과 동일한 입력/출력을 가진다.
- 유효성 검사 실패, 인증 실패, 대상 없음, 충돌 상황의 에러 응답이 정상 동작한다.
- 태그와 할 일의 관계가 정상적으로 생성/수정/삭제된다.

### 다음 Task로 넘어가기 전 확인사항

- Supabase에서 직접 쿼리했을 때 테이블과 FK가 정상 생성되었는가
- Postman/Insomnia 또는 동등한 도구로 핵심 엔드포인트가 호출되는가
- 인증이 필요한 API에 JWT Bearer 검증이 적용되었는가

## Task 2: 프론트엔드

### 목표

PRD의 사용자 스토리와 와이어프레임, 스타일 가이드를 반영한 React UI를 구현한다.

### 세부 작업 순서

1. 앱 쉘과 공통 레이아웃을 만든다.
2. TailwindCSS 스타일을 `docs/5-1-style-guide.md` 기준으로 적용한다.
3. 공통 컴포넌트 버튼, 입력, 카드, 칩, 배지 스타일을 정리한다.
4. 할 일 목록 화면을 구현한다.
5. 검색, 필터, 정렬 UI를 구현한다.
6. 빠른 추가 폼을 구현한다.
7. 할 일 추가/수정 패널 또는 모달을 구현한다.
8. 삭제 확인 다이얼로그를 구현한다.
9. 태그 관리 화면 또는 패널을 구현한다.
10. 로그인 화면은 서버 모드 선택용으로만 필요한 경우 구현한다.
11. 빈 상태와 오류 상태 UI를 구현한다.
12. 폼 검증과 에러 메시지 표시를 구현한다.
13. 반응형 레이아웃을 모바일/태블릿/데스크톱으로 나눈다.

### 예상 소요 시간

- 12 ~ 20시간

### 완료 기준 (Definition of Done)

- PRD의 핵심 사용자 스토리와 연결되는 화면이 모두 구현된다.
- 스타일이 스크린샷의 밝은 카드 중심 톤과 일치한다.
- 모바일과 데스크톱에서 주요 동작이 끊기지 않는다.
- 폼 입력, 모달, 리스트 카드가 일관된 디자인 시스템을 따른다.

### 다음 Task로 넘어가기 전 확인사항

- 와이어프레임의 주요 화면이 실제 UI로 구현되었는가
- `sm/md/lg`에서 레이아웃이 자연스럽게 바뀌는가
- 컴포넌트에 공통 디자인 토큰이 반영되었는가

## Task 3: 연동 & 테스트

### 목표

프론트엔드와 백엔드를 연결하고, 실제 사용 흐름을 기준으로 오류 처리와 동작 검증을 마무리한다.

### 세부 작업 순서

1. 프론트엔드에서 백엔드 API를 호출하도록 연결한다.
2. 목록 조회, 생성, 수정, 삭제, 완료 전환, 태그 관리 흐름을 연결한다.
3. 로딩/성공/실패 상태를 화면에 반영한다.
4. 입력 검증과 서버 검증 메시지를 일치시킨다.
5. 네트워크 오류와 4xx/5xx 응답 처리 UI를 만든다.
6. 인증 토큰 갱신과 로그아웃 흐름을 연결한다.
7. 핵심 사용자 스토리 기반의 수동 테스트 시나리오를 작성한다.
8. API 계약 테스트 또는 통합 테스트를 추가한다.
9. 화면별 접근성과 키보드 동작을 점검한다.
10. 목록, 필터, 검색, 정렬, 완료 전환의 회귀를 확인한다.
11. 로컬 저장 또는 서버 세션 정책이 의도대로 동작하는지 검증한다.

### 예상 소요 시간

- 8 ~ 14시간

### 완료 기준 (Definition of Done)

- 프론트엔드에서 실제 API를 호출해 Todo CRUD가 동작한다.
- 주요 오류 케이스가 사용자에게 이해 가능한 메시지로 표시된다.
- 핵심 흐름에 대한 테스트가 통과한다.
- 수동 테스트 기준으로 사용자 스토리가 끊기지 않는다.

### 다음 Task로 넘어가기 전 확인사항

- 목록 조회 후 생성/수정/삭제 결과가 즉시 반영되는가
- 필터와 정렬이 API 파라미터와 일치하는가
- 오류 응답이 UI를 깨뜨리지 않고 처리되는가

## Task 4: 배포

### 목표

프론트엔드와 백엔드를 각각 배포하고, 운영에 필요한 환경변수와 도메인 구성을 마무리한다.

### 세부 작업 순서

1. 프론트엔드를 Vercel에 배포한다.
2. 백엔드 API를 Vercel 또는 별도 서버리스/호스팅 환경에 배포한다.
3. Supabase 프로젝트의 프로덕션 환경변수를 등록한다.
4. Vercel 프로젝트별 환경변수를 설정한다.
5. CORS와 허용 도메인을 운영 도메인에 맞게 정리한다.
6. 프로덕션 빌드가 통과하는지 확인한다.
7. 배포 후 헬스체크 URL 또는 기본 응답을 준비한다.
8. 배포 로그와 오류 모니터링 기준을 정한다.
9. 필요하면 도메인 연결과 리다이렉트 규칙을 설정한다.
10. README 또는 배포 문서에 운영 URL과 설정 절차를 기록한다.

### 예상 소요 시간

- 4 ~ 8시간

### 완료 기준 (Definition of Done)

- 프론트엔드와 백엔드가 운영 환경에서 정상 접속된다.
- 환경변수 누락 없이 배포가 재현 가능하다.
- 운영 도메인에서 CORS 문제가 없다.
- 배포 후 핵심 API와 화면이 정상 작동한다.

### 다음 Task로 넘어가기 전 확인사항

- 운영 URL에서 목록 조회와 생성 흐름이 성공하는가
- 환경변수와 보안 정보가 저장소에 노출되지 않았는가
- 배포 이후 다시 배포할 때 같은 절차로 재현 가능한가

## 권장 진행 순서

1. Task 0으로 개발 환경과 저장소 구조를 고정한다.
2. Task 1에서 DB와 API 계약을 먼저 완성한다.
3. Task 2에서 화면을 구현하고 API 계약에 맞춘다.
4. Task 3에서 실제 동작을 연결하고 검증한다.
5. Task 4에서 운영 배포와 환경변수를 마무리한다.
