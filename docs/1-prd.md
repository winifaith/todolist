# Todolist App PRD

## 1. 제품 목적과 배경

Todolist App은 개인이 해야 할 일을 빠르게 기록하고, 확인하고, 완료 상태까지 관리할 수 있는 웹 애플리케이션이다.  
현재 많은 사용자는 메모 앱, 채팅, 기억에 의존해 할 일을 관리하고 있으며, 이 방식은 누락과 중복을 쉽게 만든다.  
이 제품의 목적은 할 일 입력과 완료 관리 과정을 최소 단계로 줄여, 사용자가 생각난 일을 즉시 기록하고 다시 빠르게 처리할 수 있게 하는 것이다.

## 2. 타깃 사용자

- 개인 일정과 업무를 스스로 관리하는 사용자
- 학생, 직장인, 프리랜서처럼 하루 단위 할 일 관리가 필요한 사용자
- 복잡한 프로젝트 관리 도구보다 가볍고 빠른 할 일 관리가 필요한 사용자
- 데스크톱과 모바일 브라우저를 함께 사용하는 사용자

## 3. 핵심 사용자 스토리

### 3.1 할 일 추가

나는 개인 사용자로서, 지금 떠오른 할 일을 빠르게 입력하고 싶다.  
그래야 잊기 전에 바로 기록할 수 있다.

Acceptance Criteria:

- 사용자는 제목을 입력해 할 일을 추가할 수 있다.
- 사용자는 마감일, 우선순위, 태그를 선택적으로 추가할 수 있다.
- 필수 입력값이 없으면 저장되지 않는다.
- 추가한 할 일은 즉시 목록에 표시된다.

### 3.2 할 일 수정

나는 개인 사용자로서, 기존 할 일의 내용을 나중에 수정하고 싶다.  
그래야 변경된 일정이나 내용을 반영할 수 있다.

Acceptance Criteria:

- 사용자는 기존 할 일의 제목을 수정할 수 있다.
- 사용자는 마감일, 우선순위, 태그를 수정할 수 있다.
- 수정 결과는 즉시 목록에 반영된다.
- 수정 중 취소하면 변경 내용이 저장되지 않는다.

### 3.3 할 일 완료 처리

나는 개인 사용자로서, 끝낸 일을 완료 상태로 표시하고 싶다.  
그래야 현재 해야 할 일과 끝난 일을 구분할 수 있다.

Acceptance Criteria:

- 사용자는 할 일을 완료 상태로 전환할 수 있다.
- 사용자는 완료 상태를 다시 미완료로 되돌릴 수 있다.
- 완료된 할 일은 시각적으로 구분된다.
- 완료/미완료 전환은 목록에서 바로 가능하다.

### 3.4 할 일 조회 및 필터링

나는 개인 사용자로서, 오늘 할 일과 조건에 맞는 할 일을 빠르게 찾고 싶다.  
그래야 필요한 작업만 바로 확인할 수 있다.

Acceptance Criteria:

- 사용자는 전체 할 일 목록을 볼 수 있다.
- 사용자는 완료 여부, 마감일, 태그, 우선순위로 필터링할 수 있다.
- 사용자는 검색어로 할 일을 찾을 수 있다.
- 사용자는 정렬 기준을 변경할 수 있다.

### 3.5 할 일 삭제

나는 개인 사용자로서, 더 이상 필요 없는 할 일을 삭제하고 싶다.  
그래야 목록이 불필요하게 복잡해지지 않는다.

Acceptance Criteria:

- 사용자는 할 일을 삭제할 수 있다.
- 삭제 전 확인 동작이 있다.
- 삭제 후 해당 항목은 목록에서 사라진다.

## 4. 핵심 기능과 우선순위

### P0

- 할 일 추가
- 할 일 수정
- 할 일 삭제
- 완료/미완료 전환
- 목록 조회
- 로컬 저장

### P1

- 마감일 설정
- 우선순위 설정
- 검색
- 필터
- 정렬

### P2

- 태그/카테고리
- 반복 일정
- 알림/리마인더
- 위젯

## 5. 비기능 요구사항

### 성능

- 기본 목록 화면은 1초 내에 상호작용 가능해야 한다.
- 할 일 추가, 수정, 삭제는 지연 없이 즉시 반영되어야 한다.
- 목록이 많아져도 기본 사용 흐름이 느려지지 않아야 한다.

### 보안 및 데이터

- 초기 버전은 서버 인증이 없더라도 데이터 유실이 없어야 한다.
- 개인 데이터는 브라우저 로컬 저장소에 보관한다.
- 사용자가 데이터를 직접 삭제할 수 있어야 한다.
- 외부로 개인 데이터를 전송하지 않는다.

### 접근성

- 키보드만으로 모든 핵심 작업을 수행할 수 있어야 한다.
- 포커스 상태가 명확하게 보여야 한다.
- 텍스트 대비는 가독성을 해치지 않아야 한다.

### 신뢰성

- 새로고침 후에도 데이터가 유지되어야 한다.
- 오프라인 상태에서도 조회와 기본 편집이 가능해야 한다.

## 6. 반응형 웹 UI 요구사항

- 지원 해상도: 360px 이상 모바일, 768px 이상 태블릿, 1024px 이상 데스크톱
- 모바일 브레이크포인트: 360px to 767px
- 태블릿 브레이크포인트: 768px to 1023px
- 데스크톱 브레이크포인트: 1024px 이상
- 모바일에서는 단일 컬럼 레이아웃을 사용한다.
- 태블릿 이상에서는 목록과 상세/편집 영역을 더 넓게 배치할 수 있다.
- 모든 해상도에서 주요 동작 버튼은 한 번에 접근 가능해야 한다.
- 화면 폭이 줄어들어도 검색, 필터, 입력의 핵심 기능은 유지되어야 한다.

## 7. Out of Scope

- 팀 협업과 할당 기능
- 댓글, 멘션, 실시간 공동 편집
- 외부 캘린더 및 메신저 연동
- 로그인 기반 클라우드 동기화
- 복잡한 프로젝트 관리 기능
- 간트 차트, 칸반 보드, 통계 대시보드

## 8. 성공 기준

- 사용자가 30초 이내에 첫 할 일을 추가할 수 있다.
- 사용자가 목록에서 할 일을 빠르게 완료 처리할 수 있다.
- 사용자가 다시 방문했을 때 이전 데이터가 유지된다.
- 사용자가 "가볍고 빠르다"는 인상을 받는다.

## 9. 문서를 보고 바로 만들어야 할 것

이 PRD를 기준으로 만들어야 할 것은 다음과 같다.

- 할 일을 추가, 수정, 삭제할 수 있는 웹 UI
- 완료 상태를 전환할 수 있는 목록 화면
- 로컬 저장을 통한 데이터 유지 기능
- 검색, 필터, 정렬을 포함한 목록 관리 화면
- 모바일과 데스크톱에서 동작하는 반응형 레이아웃
